## 📝 二、详细步骤说明

### 步骤 1: 需求分析

**输入**: 用户需求、产品反馈  
**输出**: 需求列表、优先级排序

**活动**:
- 理解用户需求
- 分析技术可行性
- 评估工作量
- 确定优先级（P0/P1/P2）

**产出物**:
- 需求清单
- 技术方案草稿

---

### 步骤 2: 文档编写

**输入**: 需求清单  
**输出**: 正式文档

**活动**:
- 编写 `REQUIREMENTS.md`（需求文档）
- 编写 `PROCESS.md`（流程文档）
- 更新 `README.md`（如有需要）
- 创建技术设计文档（复杂功能）

**文档模板**:
```markdown
# 功能名称

## 需求描述
...

## 技术方案
...

## 验收标准
...

## 预计工时
...
```

---

### 步骤 3: GitHub 更新

**输入**: 文档  
**输出**: 已提交的代码库

**活动**:
```bash
# 1. 切换到开发分支
git checkout develop

# 2. 创建文档分支
git checkout -b docs/requirements-v3

# 3. 添加文件
git add REQUIREMENTS.md PROCESS.md

# 4. 提交
git commit -m "docs: 添加 v3.0 需求文档和流程文档"

# 5. 推送
git push origin docs/requirements-v3
```

**验收**:
- 文档已提交到仓库
- 提交信息符合 Conventional Commits

---

### 步骤 4: Issue 创建

**输入**: 需求文档  
**输出**: GitHub Issues

**活动**:
- 在 GitHub 创建 Issue
- 使用 Issue 模板
- 添加标签（bug/enhancement）
- 关联里程碑

**Issue 模板**:
```markdown
## 任务描述
...

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 技术要点
...

## 预计工时
...

## 关联需求
REQUIREMENTS.md # 章节
```

---

### 步骤 5: 任务分配

**输入**: Issue 列表  
**输出**: 已分配的任务

**活动**:
- 评估任务依赖关系
- 确定执行顺序
- 分配执行者（自己或子 agent）
- 设置截止日期

**任务分配策略**:
| 任务类型 | 执行者 | 说明 |
|----------|--------|------|
| 简单 Bug 修复 | 自己 | 直接修改代码 |
| 复杂功能 | 子 agent | 并行开发 |
| 代码审查 | 子 agent | 独立审查 |
| 测试验证 | 子 agent | 自动化测试 |

---

### 步骤 6: 开发实现

**输入**: Issue、技术方案  
**输出**: 功能代码

**活动**:
```bash
# 1. 创建功能分支
git checkout -b feature/BUG-001-bridge-nodejs

# 2. 开发实现
# ... 编写代码 ...

# 3. 本地测试
npm test

# 4. 提交代码
git add .
git commit -m "feat: 实现 XXX 功能"

# 5. 推送分支
git push origin feature/BUG-001-bridge-nodejs
```

**开发规范**:
- 遵循代码规范
- 编写单元测试
- 更新相关文档
- 保持提交原子性

---

### 步骤 7: 本地测试

**输入**: 功能代码  
**输出**: 测试报告

**活动**:
- 运行单元测试
- 运行集成测试
- 手动测试核心功能
- 检查性能指标

**测试清单**:
```markdown
## 功能测试
- [ ] 核心功能正常
- [ ] 边界条件处理
- [ ] 错误处理正常

## 性能测试
- [ ] 启动时间 < 5 秒
- [ ] 内存占用 < 300MB
- [ ] CPU 占用 < 10%

## 兼容性测试
- [ ] Windows 10/11
- [ ] 不同分辨率
```

---

### 步骤 8: PR 提交

**输入**: 功能分支、测试报告  
**输出**: Pull Request

**活动**:
- 在 GitHub 创建 PR
- 填写 PR 描述
- 关联 Issue
- 请求审查

**PR 模板**:
```markdown
## 变更描述
...

## 关联 Issue
Closes #123

## 测试报告
- 单元测试：✅ 通过
- 集成测试：✅ 通过
- 手动测试：✅ 通过

## 截图（如适用）
...

## 检查清单
- [ ] 代码符合规范
- [ ] 已添加测试
- [ ] 文档已更新
```

---

### 步骤 9: 代码审查

**输入**: PR  
**输出**: 审查意见

**活动**:
- 审查代码质量
- 检查测试覆盖
- 验证功能实现
- 提出改进建议

**审查要点**:
```markdown
## 代码质量
- [ ] 代码规范
- [ ] 命名清晰
- [ ] 注释充分

## 功能完整性
- [ ] 需求已实现
- [ ] 边界条件处理
- [ ] 错误处理

## 测试
- [ ] 单元测试充分
- [ ] 测试用例覆盖边界
```

**审查结果**:
- ✅ 通过 → 进入步骤 10
- ❌ 不通过 → 返回步骤 6 修改

---

### 步骤 10: 合并发布

**输入**: 已审查的 PR  
**输出**: 已合并的代码、发布版本

**活动**:
```bash
# 1. 合并到 develop
git checkout develop
git merge feature/xxx

# 2. 运行 CI/CD
# GitHub Actions 自动执行

# 3. 创建发布版本
git tag -a v3.0.0 -m "Release v3.0.0"
git push origin v3.0.0

# 4. 构建发布产物
npm run build
```

**发布清单**:
- [ ] 代码已合并
- [ ] 标签已创建
- [ ] 构建产物已生成
- [ ] 更新日志已更新
- [ ] 文档已更新

## 🤖 三、子 Agent 协作流程

### 3.1 何时使用子 Agent

| 场景 | 使用子 Agent | 说明 |
|------|-------------|------|
| 复杂功能开发 | ✅ | 并行开发，提高效率 |
| 代码审查 | ✅ | 独立视角，更客观 |
| 测试验证 | ✅ | 自动化测试 |
| 简单 Bug 修复 | ❌ | 自己快速处理 |
| 文档编写 | ❌ | 自己把控质量 |

### 3.2 子 Agent 任务分配

```python
# 示例：使用 sessions_spawn 创建子 agent
sessions_spawn(
    task="负责 BUG-001 的 Bridge 服务 Node.js 重写",
    agentId="fullstack-dev",  # 或指定其他 agent
    mode="run",  # 一次性任务
    timeoutSeconds=3600  # 1 小时超时
)
```

### 3.3 子 Agent 管理

**任务分配**:
- 明确任务目标
- 提供技术文档
- 设置验收标准
- 指定截止时间

**进度跟踪**:
- 定期检查进度
- 查看子 agent 日志
- 必要时介入指导

**结果验收**:
- 验证功能实现
- 检查代码质量
- 运行测试用例

## 📊 四、质量管理

### 4.1 代码审查标准

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 功能完整性 | 30% | 需求 100% 实现 |
| 代码质量 | 25% | 符合规范、可读性好 |
| 测试覆盖 | 25% | 单元测试 > 80% |
| 文档完整 | 10% | 注释、文档齐全 |
| 性能优化 | 10% | 无明显性能问题 |

### 4.2 Bug 分级

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| P0 - 严重 | 系统崩溃、数据丢失 | 24 小时内 |
| P1 - 高 | 核心功能不可用 | 3 天内 |
| P2 - 中 | 非核心功能问题 | 1 周内 |
| P3 - 低 | 体验优化、UI 问题 | 2 周内 |

### 4.3 测试策略

**单元测试**:
- 覆盖核心函数
- 边界条件测试
- 错误处理测试

**集成测试**:
- API 接口测试
- 数据库操作测试
- 组件交互测试

**端到端测试**:
- 核心业务流程
- 用户场景模拟
